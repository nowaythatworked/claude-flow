# No Silent Divergence From Plan

Once a plan is agreed:
- Implement exactly what was discussed
- If you encounter something unexpected (broken assumption, new constraint, alternative approach) — STOP and surface it to the user
- Don't silently change scope, approach, or business logic

What counts as divergence (must discuss):
- Choosing a different technical approach than planned
- Changing a business logic assumption
- Adding or removing scope
- Discovering the plan won't work as expected

What's fine to decide autonomously:
- Variable/function naming
- Import organization
- Minor code style choices
- Implementation details within the agreed approach

When the user specifies HOW to do something (worktree, playwright, SST context, specific SDK) — use THAT method exactly.
